# 参与贡献

感谢你对本书的兴趣。以下是参与内容改进的流程和规范。

## 快速开始

1. 安装 [mdBook](https://github.com/rust-lang/mdBook)（建议使用最新稳定版）
2. 克隆仓库后进入项目目录，运行：

```bash
mdbook serve
```

然后在浏览器打开 `http://localhost:3000` 预览效果。

确认无误后构建最终版本：

```bash
mdbook build
```

构建产物在 `./book/` 目录。

## 内容规范

所有内容修改必须通过 `polish/polish_rule.md` 的检查项。核心要求摘要：

- **P0（底线）**：操作准确、安全提示到位、每章有阅读前提、概念有闭环
- **P1（体验）**：用"你"对话、无 emoji、术语首次出现时解释、引号统一用中文弯引号 `""`
- **P2（质量）**：章节长度合理、链接路径有效、比喻不与正文矛盾

完整规则见 `polish/polish_rule.md`。

## 命名与格式规则

### 新增章节

- 文件名使用中文，如 `新章节名.md`
- 章节标题格式：`# 章节名：副标题说明`
- 每章开头必须有 `> 阅读前提：` 引用块
- 新增章节后更新 `src/SUMMARY.md`

### 跨章节链接

- 链接文本统一用书名号包裹，如 `[《电脑配置详解》](../扩展知识/电脑配置详解.md)`
- 使用相对路径
- 链接文本必须与目标文件的实际标题一致

### 平台举例原则

正文中的产品举例遵循"方法分类优先、具体产品为辅"的原则：
- 先给出功能类别（如"AI 助手""云盘服务""搜索引擎"）
- 再用"例如"或括号补充 1-2 个代表性产品
- 不堆砌多个同类产品名（避免"DeepSeek、腾讯元宝、豆包"这种列举）
- 软件推荐清单（附录类）不受此限，可以列出具体产品

## 提交前自检

提交 Pull Request 前，请确认：

- [ ] `mdbook build` 构建成功，无报错
- [ ] 所有新增的跨章节链接指向真实存在的文件
- [ ] 通过 `polish/polish_rule.md` 的 P0 和 P1 检查项
- [ ] 无 emoji、无装饰性符号
- [ ] 中文引号统一使用 `""`
- [ ] 新概念首次出现时有解释

## 反馈方式

- 内容勘误或小修正：直接提交 PR 即可
- 较大的改动建议：先开 Issue 讨论，达成共识后再动手
- 如果不确定某处是否该改，Issue 里问即可

## Issue 模板

仓库已提供 Issue 模板：
- **内容勘误**：用于报告事实性错误、过时信息、操作步骤偏差
- **改进建议**：用于提出新的内容方向、结构调整、表达优化建议

选择合适的模板填写，能帮助我们更快处理。
